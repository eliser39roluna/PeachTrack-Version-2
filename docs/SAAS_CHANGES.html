📄 SaaS Upgrade Report

PeachTrack SaaS Transformation

A full breakdown of every change made when upgrading PeachTrack from a single-company app to a multi-tenant SaaS platform.

📅 March 2026
💻 PHP 8.4 • MySQL 8.0 • Apache 2.4
🔧 Multi-tenant • Role-based • Company-scoped

⇄ Old System vs. New SaaS

⚠ Original System

  • Hard-coded for a single business (Fuzzy Peach Salon)
  • All employees shared one database with no isolation
  • No company concept — one flat employee list
  • Only 2 roles: Manager (101) and Employee (102)
  • Login page branded specifically for one company
  • Username must end with @peach
  • No platform-level oversight or admin
  • Any user could try to reach any page
  • Employee data mixed across all managers
  • No way to onboard a new business

✓ New SaaS System

  • Generic platform — any type of business can use it
  • Each company’s data is fully isolated via Company_ID
  • New company table with name, email, timestamp
  • 3 roles: Super Admin (100), Manager (101), Employee (102)
  • Login page is platform-branded with feature highlights
  • Usernames are free-form, no forced suffix
  • Super Admin panel: KPI dashboard, company list, activate/deactivate
  • Every page checks role AND company ownership
  • Employees, shifts, tips, payroll all scoped per company
  • Super Admin can register new companies from the panel

🗃 Multi-Tenant Architecture

company_id, role, name

MySQL DB
shared + scoped

Every SQL query on pages like manage_users, shifts, payroll and reports includes WHERE Company_ID = ? bound to the session value — so Company A can never see Company B’s data.

Design Pattern Used: Shared Database, Shared Schema with Tenant Column (Option A)
All companies live in the same MySQL database. A Company_ID foreign key on the employee table acts as the tenant discriminator. Every query is filtered by the company_id stored in the PHP session at login time.

👤 Role System

Type_Code 100

Super Admin

  • Access superadmin.php dashboard
  • See all companies and platform-wide KPIs
  • Register new companies
  • Activate / deactivate companies
  • Not tied to any specific company
Type_Code 101

Manager / Admin

  • Full access to their company’s data only
  • Create & manage employees
  • Create shifts, log tips
  • View reports and payroll
  • Can view-as any employee in their company
Type_Code 102

Employee

  • View their own shifts only
  • See their own tip history
  • No access to payroll or reports
  • Cannot create/edit other employees

🔑 Login & Session Flow

1
User submits username + password login.php queries employee JOIN credential JOIN company where User_Name = ? and Is_Active = 1.
2
bcrypt verify password_verify() checks the submitted password against the stored bcrypt hash in the credential table.
3
Session is created Stores loggedin, id, name, role, company_id, company_name. Session ID is regenerated to prevent fixation attacks.
4
Role-based redirect Role 100 → superadmin.php. Role 101 or 102 → index.php (company dashboard).
5
Every page checks company scope All queries call peachtrack_company_id() to get the session company_id and bind it to every WHERE clause.

🛡 Security Improvements

AreaBeforeAfter
Data isolation No isolation — all employees in one flat table visible to any manager Company_ID scope on every query; cross-company access is impossible at the SQL level
Company registration N/A — single company, hardcoded Restricted to Super Admin only; direct URL access redirects to login
Payroll cross-access Any manager could craft a URL with another company’s employee ID payroll_details.php verifies Employee_ID AND Company_ID match
Account activation No deactivation — accounts could not be blocked Is_Active column; deactivated accounts blocked at login. Super Admin can deactivate entire companies.
Password storage bcrypt (already good) bcrypt continues; Super Admin password set via secure PHP script (not PowerShell inline to avoid $ stripping)
Session fixation Not explicitly regenerated session_regenerate_id(true) called on successful login

🌐 Public Hosting via ngrok

1
Install ngrok Downloaded to C:\ngrok\ngrok.exe from the official ngrok Windows AMD64 release. No PATH change required — the script references the full path.
2

3
PHP server binds to 0.0.0.0 Changed from localhost to 0.0.0.0:8080 so the server accepts connections from all network interfaces, not just the loopback.
4
ngrok opens public HTTPS tunnel Tunnels localhost:8080 to a public URL (e.g. https://abc123.ngrok-free.app). Accessible from any device on any network worldwide.
5

Free plan limitation: The public URL changes every time ngrok is restarted. For a permanent URL, ngrok’s paid plan or a static domain is required. The laptop must remain on and running for the site to be accessible.

🔒 March 5, 2026 — Super Admin Security & Company Status

This release adds a Change Password feature for the Super Admin account and fixes the company deactivation UI so deactivated businesses are clearly labelled and only the correct action button is shown.

📁 Modified Files

FileTypeWhat Changed
src/superadmin.php Modified Security UI Change Password: Added a change_password POST handler that verifies the current bcrypt password before hashing and saving the new one. Added a “Change Password” button above Logout in the sidebar. Clicking it opens a modal with Current Password, New Password, and Confirm New Password fields. Validation: passwords must match, minimum 8 characters, and the current password must be correct.

Company Status column: The companies SQL query now fetches COALESCE(MAX(e.Is_Active), 1) AS company_active. A new Status column in the table displays a green Active badge or a red Deactivated badge per company. Deactivated rows are visually dimmed (opacity: 0.62).

Context-aware action buttons: Active companies show only the Deactivate button; deactivated companies show only the Reactivate button (also with a confirmation prompt). Previously both buttons were always rendered simultaneously regardless of state.
scripts/start_peachtrack_ngrok.ps1 Modified Replaced all Unicode special characters (─, ═, →, —, –) with plain ASCII equivalents. PowerShell 5.1 on Windows failed to parse the script with a “string missing terminator” error caused by these characters. The fix makes the script runnable on all Windows PowerShell 5.1 environments without requiring a UTF-8 BOM or changing the execution policy.

🛡 Security Detail — Change Password Flow

1
Super Admin clicks “Change Password” in the sidebar A modal opens with three fields: Current Password, New Password, Confirm New Password.
2
Server-side validation PHP checks: new password ≥ 8 characters, new and confirm fields match, current password is correct via password_verify() against the credential table.
3
bcrypt hash stored password_hash($newPw, PASSWORD_DEFAULT) generates a new bcrypt hash. The credential table record for the Super Admin’s Employee_ID is updated.
4
Success or error feedback A styled alert banner confirms the update or explains the specific validation failure (mismatch, too short, or wrong current password).

ComponentChange

Change Password modal Frosted-overlay modal with the same fadeUp animation used by the logo modal. Fields have focus ring on the accent colour. ESC key closes the modal. Clicking outside the modal box also closes it.
Companies table — Status column New column between Total Tips and Actions. Active companies: green pill with checkmark. Deactivated companies: red pill with ×. Entire table row is dimmed at 62% opacity when deactivated for at-a-glance visibility.

🛎 March 5, 2026 — Services System

This release introduces a full Services feature. Managers can create and manage a list of billable services (with prices). Employees must select a service when logging a tip, and the sale amount is auto-filled from the service price. The Reports page now surfaces the top-performing service for the selected date range.

📁 New & Modified Files

FileTypeWhat Changed
sql/alter_services.sql New DB Creates the service table (Service_ID, Company_ID FK, Service_Name, Price, Is_Active). Adds Service_ID column to tip and a foreign key constraint fk_tip_service referencing service.Service_ID. Migration is scoped per company and safe to run once.
src/manage_services.php New UI Manager-only CRUD page for services. Supports adding a new service (name + price) and toggling active/deactivated status. The “All Services” table defaults to 5 most-recently-added rows with a live search filter and a “Show all / Show less” toggle. Newest entries are listed first (ORDER BY Service_ID DESC).
src/header.php Modified UI Added a 🛎 Services navigation link to the sidebar for users with the Manager role (role 101). Link points to manage_services.php.
src/edit_shift.php Modified UI Add Tip form now includes a required Service dropdown. Each <option> carries a data-price attribute; selecting a service auto-fills the Sale Amount field via JavaScript. The add_tip POST handler validates that service_id > 0. If no active services exist, the form is disabled and a warning is shown.
src/index.php Modified UI Log Tip form: Added a required Service dropdown with the same price auto-fill behaviour. The submit_tip POST handler now includes Service_ID in the INSERT using peachtrack_has_column() for graceful degradation on schemas without the column.

Recent Tips card: Revamped from grouped per-date sub-tables into a single flat, searchable table. New columns: Date, Time, Service, Tip Amount, Sale Amount, Method. Defaults to 5 rows with a “Show all / Show less” toggle and a live search input (filters by date, service name, and payment method).
src/reports.php Modified UI Replaced the “Selected employee” KPI card with a Top Service card. Queries service via JOIN, groups by Service_ID, orders by total tips, and displays the #1 service name with its aggregated tip total for the selected date range. Falls back gracefully if the migration has not been run.

🔄 Service Selection Flow

1
Manager creates services Via manage_services.php — enters a service name and default price. Services are scoped to the manager’s company.
2
Employee opens Log Tip or Add Tip form The Service dropdown is populated with all active services for their company. The field is required; the form cannot be submitted without a selection.
3
Sale amount is auto-filled Selecting a service instantly fills the Sale Amount field with the service’s default price via a JavaScript change listener on the dropdown. The employee can still edit the amount before submitting.
4
Tip saved with Service_ID The server validates service_id > 0 and stores it in tip.Service_ID. The Recent Tips card and Reports page then reflect the service name.
